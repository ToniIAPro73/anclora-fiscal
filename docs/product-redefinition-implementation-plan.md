## Hallazgos abiertos y notas de alcance (no accionables en FASE 00)

- **`order_lines` (Fase 02/04):** confirmado por grep directo sobre `packages/db/migrations/0000-0010` y `packages/db/src/schema.ts` — no existe tabla `order_lines` ni columna equivalente. Un pedido Shopify multi-artículo con líneas de distinta naturaleza fiscal (p. ej. ebook + físico en el mismo pedido) sigue modelado como una única fila de `commercial_orders`. Este es el hallazgo que el doc de redefinición señala como "importación de una sola fila por `Name`" — **no resuelto**, se registra como entrada para Fase 02.
- **Conflicto potencial con Fase 5b / commit `eb6e610`:** el commit `eb6e610` ("automate invoicing and add operational filters", `operation-filters.tsx`, `operations-controller.ts`) y la migración `0010_buyer_contact_evidence.sql` (facturación automática de Shopify) operan sobre `commercial_orders`/`canonical_operations` sin un modelo de `order_lines`. Si Fase 02 introduce `order_lines` como fuente de verdad por línea, ambos flujos (filtros de operaciones, facturación automática) necesitarán revisión de migración de datos. Se deja como pregunta abierta dirigida explícitamente a Fase 02, sin acción en FASE 00.
- **`docs/known-limitations.md` desactualizado:** el documento (líneas ~19-22) afirma que "Operaciones, Conciliación, Facturación, VERI\*FACTU y Expedientes IVA son vistas de demostración sobre casos fijos... no usan repositorios persistentes". Esto es **falso** a fecha de esta fase — contradicho por los servicios ya cableados (`matching-service.ts`, `tax-decision-service.ts`, `ingestion-normalization-service.ts`, commits `a7e6b5f`→`145a5b4`). Ver nota de corrección en `docs/implementation-phase-log.md` (sección "limitaciones abiertas" de la entrada FASE 00). La corrección del propio archivo se deja para una fase de docs dedicada, no FASE 00, para no exceder el alcance "docs de trazabilidad + ADR + test" definido en el master prompt para esta fase.
